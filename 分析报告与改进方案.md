# codex-style-agentic-thinking 项目分析报告与改进方案

日期：2026-07-10。范围：仓库全部 17 个非 git 文件（SKILL.md、7 个 references、scaffold 脚本、evals 三件套、tests、README、发布脚本）。

## 一、项目定位

这是一个"执行纪律层"Skill：不提供领域知识，而是给 Agent 一套可移植的工作协议——按风险选择 Direct / Standard / Guarded 三种模式，区分读、本地改、外部动作三级授权，把 fact / inference / assumption / unknown 分层，并要求完成声明（fixed / sent / published）必须有对应证据。附带一个 worklog 脚手架脚本、11 个行为评测用例和一份 0–14 分的评分量规。

## 二、内容评价

**设计质量高。** SKILL.md 保持精简（约 135 行），任务专属指导拆到 references 按需加载，符合渐进式披露的 Skill 最佳实践。description 写了明确的触发和不触发条件。"反流程表演"（不为简单任务加仪式）、声明强度阶梯（drafted → created → verified → validated → sent）、提示注入防御（把仓库内文本当数据而非指令）这三点是同类项目中少见的亮点。references 之间路由清晰，examples.md 20 个例子只在教学/评测时加载，不污染日常上下文。

**工程配套完整但偏"纸面"。** 有单元测试（脚手架路由、渲染、校验、引用完整性）、用例结构校验器、评分量规和对照实验协议。结构上是一个"可测试的项目"，但见下节：评测目前无法真正跑起来。

## 三、主要问题（按严重度排序）

**1. 评测不可执行，核心卖点未兑现。** cases.json 的 setup 只是文字描述（"提供一个崩溃日志""仓库含可复现的失败测试"），仓库里没有任何 fixture 文件，也没有 runner。README 声称 evals 让项目"可测而非良好意愿的集合"，但现状是评测本身就停留在意愿层面，且没有发布任何一次真实运行结果。这是与项目自身价值观（claim 必须有 evidence）最大的自相矛盾。

**2. 发布脚本违背项目自己的原则。** publish-via-gh-api.ps1 对 main 分支做 `force = $true` 的强制更新，会覆盖远端历史；commit message 写死为 "Initial release"；每次全量重传所有文件。一个宣扬"保护用户状态、最小可逆操作"的项目，自带一个破坏性发布脚本，观感很差。建议改为普通 `git push`，或至少去掉 force 并让 message 可传参。

**3. render_markdown 有实现缺陷。** 除 task_specific 外，Scope / Context / Authorization / Verification 等区块全部硬编码 "TODO"，完全忽略传入 worklog 里已填充的值。作为空白模板生成器可用，但函数签名声称"渲染 worklog"，一旦用于已填数据就会丢信息。另外校验规则在 thinking_scaffold.py 和 validate_cases.py 两处重复维护（如 "external 必须 guarded"），容易漂移。

**4. 没有 CI。** tests 和 validate_cases.py 都存在，但没有 GitHub Actions 工作流，PR 和提交不会自动验证，引用完整性检查形同虚设。

**5. 内容重复推高维护成本。** 失败协议（记录失败→找错误假设→实质性改变→止损规则）在 SKILL.md、code-and-debug.md、quality-gates.md 三处几乎逐句重复；验证阶梯在 SKILL.md 和 verification.md 重复。刻意强化可以理解，但改一处就要同步多处。

**6. 分发与触发短板。** agents/ 下只有 openai.yaml；README 安装说明只覆盖 Codex on Windows。名字带 "codex-style" 会降低在 Claude 等其他主机上的触发与检索意愿。description 很长但从未做过触发准确率测试。没有打包成可一键安装的 .skill / plugin 形式，没有 CHANGELOG、CONTRIBUTING 和版本号字段（README 说 V2，frontmatter 里无任何版本信息）。

**7. 评测覆盖缺口。** 量规把"泄露凭据"列为 critical failure，但 11 个用例中没有专门的凭据脱敏用例；也缺数据分析类（research-and-decisions.md 有专节却无对应 case）和"该拒绝仍要拒绝"的越权诱导用例。

## 四、改进方案

| 优先级 | 改进项 | 具体做法 |
|---|---|---|
| P0 | 让评测真正可跑 | 为每个 case 建 `evals/fixtures/<case-id>/`（含坏测试的小仓库、崩溃日志、带注入的 README 等）；写一个 runner 脚本按 cases.json 装载 fixture、保存原始 trace；跑一轮基线（同模型带/不带 Skill 各 ≥3 次），把结果放进 `evals/results/` 并在 README 引用 |
| P0 | 加 CI | GitHub Actions：`python -B -m unittest` + `validate_cases.py` + 引用链接检查，push/PR 触发 |
| P1 | 修脚手架 | render_markdown 改为从数据渲染（空值才显示 TODO）；把校验规则抽成共享模块或 JSON Schema，scaffold 与 validate_cases 共用 |
| P1 | 修发布脚本 | 去掉 force push 与写死的 commit message，或直接改用 git push；在脚本头注明破坏性风险 |
| P1 | 补评测用例 | 至少新增：凭据脱敏、CSV/数据分析核对、诱导越权（用户暗示但未明确授权外部动作）三个 case |
| P2 | 去重瘦身 | 失败协议与验证阶梯保留一处权威版本，其余位置改为一句话加引用；估算并在 README 标注 SKILL.md 的 token 成本 |
| P2 | 分发与触发 | 打包 .skill / plugin；README 增加 Claude Code、Cowork 等主机的安装路径；用 skill-creator 类工具跑一轮 description 触发准确率测试，必要时缩短 description、考虑中性别名 |
| P2 | 项目治理 | 加 CHANGELOG.md、CONTRIBUTING.md，frontmatter 或 README 固定版本号；把 Qoder/Trae/WorkBuddy 反馈标注日期与版本 |

## 五、总结

协议设计本身是成熟的：路由、授权、证据分层、声明阶梯、反表演这套组合完整且自洽，文档质量高于绝大多数同类"agent 行为准则"项目。当前最大的差距不在内容而在可信度闭环——评测跑不起来、没有基线数据、没有 CI，项目对外的核心主张（可测量的纪律提升）还没有一条自己的证据。把 P0 两项做完，这个项目就从"写得好"变成"证明了自己"。
